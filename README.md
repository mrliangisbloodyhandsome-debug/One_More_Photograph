每日一图 (One More Photograph)
一、 产品功能介绍
1.1 产品概述
“每日一图”是一款极简风格的图片浏览应用。旨在为用户提供快速、流畅的视觉体验，每次点击都能从云端获取一张随机的高质量风景或摄影图片，为用户带来“未知的惊喜”。
1.2 核心功能
•	一键获取： 用户无需复杂操作，点击“探索新风景”按钮即可触发刷新。
•	随机展示： 对接 Picsum 随机图片接口，保证每次展示的内容不重复。
•	加载反馈： 在图片下载过程中提供系统原生加载占位符，优化等待体验。
•	自适应显示： 图片自动适配屏幕宽度，保持良好的视觉比例。
1.3 适用场景
适用于用户碎片化时间的视觉放松，或作为图片加载技术验证的测试工具。
二、 程序概要设计
本程序采用标准的 Android 单页面应用模式开发，基于 Kotlin 语言。
2.1 用户界面设计
•	布局容器： 使用 ConstraintLayout以保证在不同尺寸屏幕上的兼容性。
•	核心控件：
o	ImageView：位于屏幕上方，用于展示网络图片。
o	Button：位于屏幕下方，作为用户交互的触发器。
2.2 逻辑流程设计
1.	初始化： App 启动时，MainActivity 加载布局，并自动触发一次点击事件以加载首张图片。
2.	交互触发： 用户点击按钮，监听器捕获事件。
3.	URL 构建： 程序生成带有随机参数的时间戳 URL（防止服务器缓存旧图）。
4.	网络请求： 调用 Glide 框架发起异步网络请求。
5.	资源渲染： 图片下载完成后，自动填充至 ImageView，若下载中则显示占位图。
三、 软件架构图
本应用遵循简单的 MVC (Model-View-Controller) 分层思想：

四、 技术亮点及其实现原理
本项目在满足基础功能的同时，重点解决了网络加载性能与用户体验两大问题。
4.1 亮点一：高性能图片加载引擎 (Glide)
•	实现原理：
原生 Android 网络请求需要在子线程下载流、手动解码 Bitmap、再切换回主线程更新 UI，极易导致内存溢出或界面卡顿。
本项目引入 Glide 4.x 库，它封装了以下复杂逻辑：
1.	异步线程池： 自动在后台线程下载图片，不阻塞主线程（UI线程），确保应用“不卡顿”。
2.	资源自动释放： 当 Activity 销毁或图片切换时，Glide 自动回收 Bitmap 内存，满足作业要求的“无明显资源泄露”。
3.	生命周期感知： 自动感知当前页面的生命周期，防止在页面关闭后继续下载造成的崩溃。
4.2 亮点二：服务端裁剪优化
•	实现原理：
原始图片通常高达 4K 分辨率（数 MB 大小），直接加载会导致下载慢且消耗大量内存。
我们在代码中使用了特定参数的 API：
https://picsum.photos/300/300
技术原理： 通过 URL 参数告知服务器仅返回 300x300 像素的缩略图。这使得数据传输量减少了约 95%，极大地提升了加载速度，在弱网络环境下也能快速加载。
4.3 亮点三：防缓存机制
•	实现原理：
Glide 默认拥有强大的缓存机制，如果 URL 不变，它会直接读取本地缓存。为了实现“每次点击都是新图片”的功能，我们在 URL 后动态拼接了时间戳：
url + "?random=" + System.currentTimeMillis()
技术原理： 这欺骗了缓存系统，使其认为每次请求的都是不同的资源，从而强制从网络重新下载新图，确保了内容的随机性。
